작동원리

그렇기 때문에 활용하고 있는게 Virtual DOM이다.

변화가 일어나면 실제 브라우저 DOM에서 뭔가를 바꾸는게 아니라, 가상의 DOM에 일단 그리고 나서, 그리고 실제 브라우저 DOM과 가상의 DOM을 비교를 해서 차이점에 따라서 변화를 일으키는 것이다.

변동이 있을 때마다 새로 그린다고 생각하면 된다. DOM에 있는 어떤 값을 어떻게 바꿀지에 대한 고민은 하지 않아도 된다. 실제 브라우저에서 작동할 때는 변화가 필요한 곳에만 반영된다.

컴포넌트란

재사용 가능한 UI이다. 단순 템플릿 이상의 구현체이다.

  • UI의 생김새를 정의한다. 내부적으로 메소드를 정의할 수 도 있다.
  • 부모에게서 값을 받아와서 적당한 곳에서 보여주거나 사용한다.
  • 자기 자신의 사용할 값을 정의하고 필요할 때 업데이트 한다.
  • UI에서 보여줘야 하는 것이 바뀌면 DOM에 변화를 준다.

컴포넌트를 만들 때는 class 형태(ES6 문법)로 만들 수도 있고 함수 형태로 만들 수도 있다.

함수형 컴포넌트와 클래스형 컴포넌트의 차이

함수형 컴포넌트는 state/LifeCycle 같은 기능이 빠져있어 아주 조금 더 가볍다. 그래서 초기 랜더링이 조금 더 빠르고 메모리 자원을 조금 덜 사용한다. 컴포넌트를 몇천개 몇만개씩 렌더링 해야하는 상황에서 만약에 단순히 돔한테서 값을 받아와서 보여주기만 하는 컴포넌트의 경우 함수형 컴포넌트를 작성하면 조금 성능이 개선될 수 있다. 대부분의 경우 성능 차이는 미세하지만 선언이 간단해서 자주 사용한다.

State

앞에서 배운 props는 부모가 자식한테 값을 넘겨줄 때 사용한다. 값을 받아와서 수정하려면 방법이 없다. 만약에 값을 수정을 하려면 값을 받아온 자식이 수정하는 게 아니라 부모가 값을 수정하는 함수를 props로 넣어준다. (추후 더 학습할 예정)

만약에 자기 자신 내부에서 사용하고 싶고, 업데이트를 함에 따라 새로 렌더링을 하고 싶은 경우에는 state를 사용한다.

state는 컴포넌트 내부에서 변경할 수 있다. 내부에서 어떤 값이 변경이 되면 다시 렌더링을 하게 해준다. 값을 업데이트 할 때는 언제나 setState라는 함수를 사용한다.

LifeCycle API (생명주기) (이미지 첨부 예정)(처음엔 안 외워지는게 당연하다!!)

리액트 컴포넌트의 생명주기에 대해서 알아보자.

브라우저에 컴포넌트가 나타나는 것을 마운트라 한다. 리액트 컴포넌트의 정보가 업데이트 되면 그것은 업데이트라 한다. 더 이상 컴포넌트가 필요없어졌을 때, 브라우저에서 컴포넌트가 사라지는 것을 언마운트라한다.

LyfeCycle API는 컴포넌트가 브라우저에서 나타날 때, 사라질 때, 그리고 업데이트 될 때 호출되는 API이다.

이 중간 과정에서 어떤 작업을 해주고 싶다면 LifeCycle API를 사용하면 된다.

컴포넌트는 일단 초기 렌더링이 되고 아래가 리렌더링 되는 경우이다.

  1. props가 바뀔 때
  2. stage가 바뀔 때
  3. 부모 컴포넌트가 렌더링 될 때
    : 부모 컴포넌트가 리렌더링 될 때 자식 컴포넌트도 렌더 함수가 호출이 된다. 컴포넌트의 개수가 몇개 되지 않으면 사실 성능적으로 부하가 되지 않는다. 버츄얼 돔에 그리는 것도 단순히 CPU가 조금 일하는 것 정도이다. 하지만, 어플리케이션의 규모가 커질 수록 부하가 생길 수도 있다. 비록 브라우저의 엔진이 일하지 안더라도 단순히 자바스크립트의 프로세서가 일하는 것 만으로도 부하가 걸릴 수 있다. 그런것을 최적화 해주는 것이 컴포넌트의 업데이트 최적화이다.
  4. tihs.forceUpdate() 함수를 통해 강제 렌더링

마운트

컴포넌트가 브라우저에 처음 나타나는 것을 말한다.

(1) constructor : 생성자 함수. 컴포넌트의 모든 기능이 잘 작동하게 끔 초기화 시켜주는 함수이다.

(2) componentDidMount() : 컴포넌트가 화면에 출력되고 난 다음에 호출되는 함수이다.

  • 외부 라이브러리 연동 : D3, masonry, etc
  • 컴포넌트에서 필요한 데이터 요청 : Ajax, GraphQL, etc
  • DOM 에 관련된 작업 : 스크롤 설정, 크기 읽어오기 등

업데이트

props/state가 바뀌거나 부모 컴포넌트가 리렌더링 되거나 강제 업데이트 할 때를 말한다.

(1) static getDerivedStateFromProps(nextProps, prevState) : props로 받아온 것을 state로 동기화 시켜야 할 경우 사용한다. (setState를 하는 것이 아니라 특정 props가 바뀔 때 설정하고 싶은 state값을 리턴하는 형태로 사용한다.)

예)

(2) shouldComponentUpdate(nextProps, nextState) : 리액트 성능 최적화에 있어 중요한 함수이다. 렌더하는 것을 방지해주는 함수로, true를 리턴하면 일반적인 방법으로 렌더 함수를 버츄얼 돔에 그린다. 이 함수를 구현하지 않으면 기본적으로 true를 리턴 하도록 되어있다. 너무 자주 업데이트가 되는 컴포넌트 경우 최적화가 필요한데, 특정 조건에 따라 false를 리턴해주면 버츄얼 돔에 모든 것을 그리는 과정에서 해당 컴포넌트는 렌더를 호출 하지 않고 기존에 사용하던 것을 그대로 사용하도록 한다.

Virtual DOM에 렌더링 하는 것을 아끼면 성능을 최적화 할 수 있다!

(3) getSnapshotBeforeUpdate(prevProps, prevState) : DOM 업데이트가 일어나기 바로 직전의 DOM의 상태를 가져오는 용도이다. 여기서 리턴 하는 값은 componentDidUpdate에서 3번째 파라미터(snapshop)로 받아 올 수 있다. (강사님이 보여준 예는 스크롤를 원하는 위치에 고정 시키는 예, 이런 용도 빼고는 실제로 개발하면서 사용하고 싶다고 생각한 적이 없다고 함.)

(4) componentDidUpdate(prevProps, PrevState, snapshos) : 컴포넌트가 업데이트 되서 리렌더링을 마치고 난 시점에 호출된다. 차트를 만들었다고 생각해보자. 차트는 props나 state의 값에 따라 그려진다. componentDidMount에서 특정 데이터를 가지고 차트를 만들었다고 가정하자. props나 state이 값이 바뀌면, componentDidMount는 가장 처음에만 호출 되고 자동 호출되지 않기 때문에 외부 라이브러리도 에게도 값의 변경을 알려야 한다. 그걸 하기 위해 사용하는 것이 componentDidUpdate이다.

언마운트

컴포넌트가 더 이상 필요하지 않게 되면 단 하나의 API가 호출된다

componentWillUnmount() : 이벤트, setTimeOut, 외부 라리브러리 인스턴스 제거하는데 사용된다.

*참고

componentDidCatch(error, info) : render에서 에러 발생 감지 역할을 한다. 자신 말고 자신의 자식 컴포넌트에서 에러가 발생 했을 때 이를 감지해낼 수 있다.
