한 일

  • Spring-Boot를 이용한 QnA Board 만들기 반복주기 6까지 드디어 마쳤다. 진짜 오래 했다.(휴.반성의 연속) 6번째 반복주기 마지막에 쉘 스크립트를 이용해 배포 자동화를 구현 하는데 이것은 포비 말처럼 참 짜릿한 경험이다. 이번 학습을 통해서 배운거는 Spring-boot, JPA 사용이 얼마나 편리한지이다. 작성한 코드들을 다시 살펴보고 학습한 내용을 정리하자.

정리

  • ORM : Object-relational mapping = 객체와 관계(관계형 데이터베이스)와의 설정. RDB 테이블을 객체지향적으로 사용하기 위한 기술이다. RDB 테이블은 객체지향적 특징(상속, 다형성, 레퍼런스, 오브젝트 등)이 없어 자바와 같은 언어로 접근하기 쉽지 않다. ORM을 사용하면 오브젝트와 RDB사이에 존재하는 개념과 접근을 객체지향적으로 다룰 수 있다.
  • JPA : Java Persistent API. JPA는 ORM 표준기술로 Hibernate, OpenJPA, EclipseLink, TopLinkEssentiols와 같은 구현체가 있고, 표준 인터페이스가 JPA이다.
  • 간단한 입력, 조회 수정(CRU)을 한다고 가정 해보자.
    • 일단 JDBC가 제일 불편하다. 쿼리를 생성, 실행, 데이터 매핑을 반복적으로 해야하고 connetion, close코드도 반복적으로 등장한다.
    • MyBatis의 경우 connetion, close, parameter mapping, data mappaing은 위임되지만 여전히 쿼리문을 작성을 반복해야 한다.
    • JPA는 connection, close 위임하고 쿼리를 자동 생성하고 실행한다.
    • https://www.slideshare.net/ssusere4d67c/jpa-53004111 (참고, 우앙 slipp..)
  • 학습하면서 사용한 어노테이션들…에 대해 정리해보자. spring JDBC나 MyBatis를 재학습하고 나면 더 이해하기 쉬울거 같긴하지만 나중을 다시보기 위해서라도 정리
    • @Entity : JPA가 관리할 객체. 해당 클래스가 엔티티임을 알리기 위해 사용한다.
    • @Column : 필드와 테이블의 컬럼을 맵핑시켜준다. 생략시 필드의 이름이 컬럼의 이름으로 자동 매핍이 된다.
      • name(String) / nullable(boolean) / length(int) / unique(boolean) / insertable(boolean) / updatable(boolen)
    • @Id
    • @GeneratedValue
    • @RestController
    • @JsonProperty : JSON으로 변환시 필욯하다.
      • @JsonProperty(“name”) : JSON으로 변환시 속성이름을 변경 할 수도 있다.
    • @JsonIgnore : JSON으로 변환시 무시한다.
    • @ManyToOne
    • @OneToMany
    • @JoinColum
    • @Lob
    • @MappedSuprClass
    • @EntityListeners(AuditingEntityListener.class)
