(부스트코스) Part2 Maven

2018-05-02

프로젝트가 크고 복잡해 질수록 사용하는 라이브러리가 많아지고 관리도 어려워진다. 소스를 컴파일하고 배포하는 것도 점점 어려우진다. 이러한 문제를 해결하기 위해 다양한 도구들이 존재하는데 그중에 하나가 Maven이다.

Maven이란?

Maven을 사용하면 빌드, 패키징, 문서화, 테스트와 테스트 리포팅, git, 의존성 관리, svn등과 같은 형상 관리 서버와 연동(SCMs), 배포 등의 작업을 손쉽게 할 수 있다.

Maven을 이해하려면 CoC(Convention over Configuration)이라는 단어를 먼저 이해해야 하는데, CoC란 일종의 관습으로, 예를 들어 프로그램의 소스파일은 어떤 위치에 있어야 하고 소스가 컴파일 된 파일은 어떤 위치에 있어야 하고 등을 미리 정해놓은 것이다.

Maven을 사용할 경우 이점?

  • Maven을 사용하면 직접 다운로드 하지 않고 설정파일에 몇 줄 적는 것만으로 라이브러리를 사용할 수 있다.
  • Maven에 설정된 대로 모든 개발자들이 일관된 방식으로 빌드를 수행할 수 있다.
  • 다양한 플러그인을 제공해주므로, 굉장히 많은 일들을 자동화 시킬 수 있다.

Maven 기본

Archetype을 이용하면 Maven 기반 프로젝트를 생성할 경우 생성된 프로젝트 하위에 pom.xml 파일이 생성된다.

각 태그의 의미

  • project : pom.xml 파일의 최상위 루트 엘리먼트 이다.
  • modelVersion : POM model의 버전이다.
  • groupId : 프로젝트를 생성하는 조직의 고유아이디를 결정한다. 일반적으로 도메인 이름을 거꾸로 적는다.
  • artifactId : 해당 프로젝트에 의해 생성되는 artifact의 고유 아이디를 결정한다. Maven을 이용하여 pom.xml을 빌드할 경우 다음과 같은 규칙으로 artifactId가 생성된다. artifactid-version.packaging. 위 경우 빌드할 경우 mysamexamles-1.0-SNAPSHOP.jar파일이 생성된다.
  • packaging : 해당 프로젝트를 어떤 형태로 packaging 할지 결정한다. jar, war, ear 등이 해당 된다.
  • version : 프로젝트의 현재 버전. 프로젝트가 개발중일때는 SNAPSHOP을 접미사로 사용한다. Maven은 버전 관리 기능은 라이브러리 관리를 편하게 한다.
  • name : 프로젝트의 이름이다.
  • url : 프로젝트 사이트가 있따면 사이트 URL을 등록하는 것이 가능하다.

Maven을 이용할 경우 얻게 되는 큰 이점중 하나는 Dependency Management기능이다. pom.xml에서 dependencies 엘리먼트각 Dependency Management기능의 핵심이라고 할 수 있다. 해당 엘리먼트 안에 필요한 라이브러리를 지정할 수 있다.
