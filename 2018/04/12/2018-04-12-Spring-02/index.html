PART 1 프로젝트의 기본 구조 구성

6. 스프링 + MyBatis

MyBatis는 JDBC에서 개발자가 직접 처리하는 PreparedStatement의 ? 에 대한 설정이나 ResultSet을 이용한 처리가 이루어지기 때문에 기존 방식에 비해 생산성이 좋아진다.

MyBatis의 이전 버전인 iBatis는 개발자가 모든 SQL을 XML로 작성하고, SQL 문을 사용하는 DAO 클래스를 설계해서 SQL문을 호출하는 방식의 코드를 작성했지만, MyBatis는 애노테이션을 지원하고 인터페이스와 애노테이션을 통해 SQL문ㅇ르 설정하고 처리할 수 있는 형태로 발전했다.

MyBatis를 이용할 때 SQL문을 사용하는 방식은 크게 아래와 같이 나눠진다.

  1. XML만을 사용해 SQL문을 설정, DAO에서 XML을 찾아서 실행하는 코드를 작성하는 방식

    • 장점 : SQL문의 수정, 유지보수에 적합
    • 단점 : 코드 양이 많아지고 복잡성 증가

  2. 애노테이션과 인터페이스만을 이용해 SQL을 설정

    • 장점 : DAO 없이도 개발이 가능하기 때문에 생산성 증가
    • 단점 : SQL문을 애노테이션으로 작성하므로, 수정이 일어나는 경우 매번 다시 컴파일 해야함

  3. 인터페이스와 XML로 작성된 SQL문의 활용

    • 장점 : 간단한 SQL을 애노테이션, 복잡한 SQL문읜 XML로 하는등의 유연한 처리 가능
    • 단점 : 개발자의 방식에 차이가 있을 수 있기 때문에, 유지보수가 중요한 프로젝트의 경우에 부적합

국내의 대부분의 경우는 1번에 속한다.(라고 책에서…)

MyBatis를 XML을 사용해 작성하는 경우 코딩의 순서

  1. 테이블 생성 및 개발 준비

  2. 테이블 생성 및 기타 데이터 베이스 관련 설정

    • 테이블 명은 일반적으로 tbl_ 로 시작하는 것이 좋다. (DB내 여러 종류의 객체와의 혼란을 피하기 위해서)

  3. 도메인 객체의 설게 및 클래스 작성

    • 개발을 할때 가장 중요한 용어가 될만한 명사를 흔히 도메인(domain)이라고 한다.

  4. DAO 인터페이스의 작성

    • 일반적인 경우 dao 혹은 persistence라는 패키지에 작성한다.

  5. 실행해야 하는 기능을 인터페이스로 정의

    • 향후 DB관련 기술이 변경 되더라도, DAO만 변경해서 처리할 수 있게 하기 위해서이다.

  6. XML Mapper의 생성과 SQL문 작성

    • MyBatis에서는 SQL을 저장하는 존재를 Mapper라는 용어로 표현한다. Mapper는 XML과 인터페이스를 이용할 수 있다. (아래는 XML로 작성하는 경우)

  7. XML 작성 및 SQL 작성

    • XML로 작성되는 Mapper 파일의 경우 Java로 작성된 클래스와 경로를 분리해주는 것이 유지보수에 있어서 필수 적이다.
      1. scr/main/resources 패지지 밑에 mapper 패지키를 한개 만들자.
      2. XML Mapper는 상단 파일에서 사용하는 태그들에 대한 정보가 기록된 DTD를 추가한다. (참고)
      3. XML Mapper를 작성할 때는 namespace라는 설정에 가장 신경을 많이 써야한다.

  8. MyBatis에 작성된 XML Mapper를 인식하도록 설정

    • MyBatis가 동작하면 XML Mapper를 인식해야만 정상적인 동작이 되므로 root-context.xml의 설정을 변경한다.

  9. DAO 구현

    • DAO 인터페이스와 Mapper의 작성이 완료됐다면 실제 이를 구현하는 구현 클래스를 작성해야 한다. MyBatis에서는 SqlSesiionTemplate라는 것을 이용해서 DAO를 구현하므로, 일단 SqlSessionTemplate부터 설정한다.
    • DAO의 작업에서 가장 번거로운 작업은 DB와 연결을 맺고, 작업이 끝난 후에 연결을 close() 하는 것이다. mybatis-spring 라이브러리는 이것을 처리할 수 있는 SqlSessionTemplate를 제공한다. 이것은 MyBatis의 SqlSession 인터페스를 구현한 클래스로 기본적인 트랜잭션 관리나 쓰레드 처리의 안정성 등을 보장해주고, 데이터베이스의 연결과 종료를 책임진다.

  10. DAO 인터페이스를 구현한 클래스 작성

    • MemberDAO 인터페이스를 구현하는 클래스는 앞에 잓어된 SqlSessionTemplate를 주입 받아서 사용하도록 구성한다.
    • MyBatis의 SqlSession에는 SQL문을 처리할 수 있는 기능들이 존재한다. (참고) (insert/ update/ delete / seleteOne /selectList)

  11. 스프링 상에 DAO 등록 및 테스트

    • MemberDAOImpl이 @Repository 애노테이션이 설정되더라도 스프링이 해당 패키지를 스캔하지 않으면 제대로 스프링의 빈이 등록되지 못한다. 이처리는 root-context.html에서 한다.
    • 설정된 패키지의 MemberDAOImpl에 스프링의 빈으로 등록되었다는 아이콘이 보여지는지 반드시 확인해야 한다.

MyBatis의 #{} 문접

XML Mapper를 사용하는 경우 가장 먼저 작업하는 내용은 DAO 인터페이스 이다. (위에서 이미 작성함) 작성된 인터페이스의 메소드 이름을 기초로 src/main/resources/mapper에 XML Mapper를 작성한다.

Mybatis의 경우 기본적으로 PreparedStatement를 이용해 처리 된다. PreparedStatement에 들어가는 파라미터를 사용할 때는 #{ } 기호를 이용해서 처리한다.

  • 파라미터가 여러 속성을 가진 객체인 경우 #{num}은 getNum() 혹은 setNum()을 의미한다.
  • 파라미터가 하나이고, 기본 자료형이나 문자열인 경우 값이 그대로 전달된다.
  • 파라미터가 Map 타입인 경우 #{num}은 Map객채의 키값이 num인 값을 찾는다.
